## Detailed Problem

Lane Keep Assist (LKA) systems rely on accurate lane position estimates delivered in real time. The detection step — identifying where lanes are in an image — has improved significantly with deep learning, but detection alone is not sufficient: detected positions still contain noise, and small errors in lane localization translate directly into incorrect steering control.

Post-processing the detected lanes through a localization step is the natural solution. A prior two-stage approach frames this as a sequential decision problem: a Deep Q-Learning (DQL) agent takes the initially detected lane position and a local image crop, then moves the detection iteratively until it reaches a termination point near the ground truth. This is an attractive hypothesis because it decouples refinement from detection, and DQL's iterative nature should allow the agent to handle cases where one-shot detection fails — occlusions, ambiguous markings, curved lanes.

The study's central question was whether this hypothesis holds in practice, and specifically whether operating on Bézier curve parameters rather than individual points would improve localization by providing a more compact, noise-robust representation for the agent.

The operational environment adds constraints: experiments ran on a single NVIDIA Quadro RTX 4000 (8 GB VRAM), time for hyperparameter tuning was limited, and the reference implementation contained undocumented bugs that had to be diagnosed before any results were interpretable.

## Background

The two-stage detection-and-localization framing separates a hard problem into two tractable ones. The detection stage outputs approximate bounding boxes around each lane. The localization stage refines those detections to precise landmark positions.

The localization stage is formulated as a game: the agent controls a single landmark point, selects from four actions (move left, move right, delete, terminate), and receives a reward based on whether the move brought the point closer to or further from the ground truth. The agent's state is the current landmark position concatenated with features extracted from a local image crop around the current position.

The hypothesis motivating curve extension: operating on individual points makes the problem unnecessarily high-dimensional. A lane with 50 detected points requires the agent to iterate over all 50 sequentially; replacing them with 3 Bézier control points reduces the representation to 3 positions while preserving the lane's geometric shape. This should reduce the agent's action space and focus learning on the curve's curvature parameters rather than individual noisy point adjustments.

## Architecture

The pipeline has three software modules operating in sequence.

**Lane Detector** — CV-based using OpenCV's `findContours` with Hough Transform. The detector outputs a bounding box around each lane, which is cropped and passed to the localizer. The slope of the predicted lane serves as the initialization for localization. (An alternative DL-based detection path exists in the code but was not the focus of this study.)

**Lane Localizer** — The core DQL module. Two agent classes implement the localization game:

- `PointAgent`: controls a single landmark point; 4 actions; operates across all lane points sequentially
- `CurveAgent`: controls the curvature control points of a 3-point Bézier curve; 9 actions (2D directional moves + terminate)

Both agents share the same underlying DQN architecture: a pretrained ResNet-50 feature extractor (frozen) followed by two FC layers, with a final output layer matching the action count. The image input is resized to 224×224 to satisfy ResNet-50's minimum input size.

**Lane Evaluator** — A TuSimple-compatible evaluation module adapted from the dataset's official evaluation code. Computes accuracy, FPR (false positive rate — detected lanes where there are none), and FNR (false negative rate — missed lanes) by comparing reconstructed full-image lane predictions against ground truth.

## Data

TuSimple is a highway driving dataset of 6,408 images at 1280×720 resolution, split into 3,626 training, 358 validation, and 2,782 test images. Lanes are annotated as sequences of x-coordinate positions at fixed y intervals. The dataset is predominantly straight highway lanes — a property that turned out to matter for the curve localizer results, discussed in [Results](#results).

For DQL training, each lane in the dataset is processed into a labeled crop: the detector identifies the lane region, the crop is resized to 224×224, and the localization game is run on the resulting image with the detected point positions as the initial state. Batch size is 1 (one lane crop per training step), consistent with the online nature of RL training.

## Engineering Decisions

### Diagnosing 14 implementation bugs in the reference code

Before any experiments could be trusted, the reference implementation had to be audited. Examining the codebase against the paper description revealed 14 discrepancies. The five most structurally significant — numbered here because three of them were present only in the code, not described in the paper:

1. **Incorrect Bellman's equation** — Q-values were updated using the best *action* rather than the Q-value *corresponding to* the best action. The network was trained toward incorrect targets throughout.
2. **MSE loss applied to the wrong quantity** — Loss was computed as the difference between Q-values rather than between predicted and target Q-values. The paper describes standard MSE; the code implements something different.
3. **No exploration** — Epsilon-greedy was not implemented. The agent always exploited its current (untrained) estimates, preventing it from learning from diverse states during warm-up.
4. **Manual termination** — Terminal actions were forced externally rather than learned, preventing the agent from discovering optimal stopping points.
5. **Replay memory pre-filled at ε=1.0** — Pre-filling entirely with maximum-exploration actions populates the buffer with outdated, suboptimal experiences that persist through training.

Issues 1, 2, and 5 were not described in the paper — they appear only in the code. The remaining nine issues covered data preprocessing errors and additional Q-value calculation problems. The corrected implementation diverges substantially from the published methodology while preserving the core DQL localization framing.

### Switching to D-DQN with prioritized experience replay

After fixing the base bugs, the single-network variant was replaced with Double Deep Q-Network (D-DQN). D-DQN separates the action-selection network (policy network, updated every step) from the value-estimation network (target network, updated with soft updates at τ=0.005). This reduces Q-value overestimation, which is especially important in dense lane prediction tasks where the reward signal is noisy and correlated.

Prioritized experience replay was added to weight high-TD-error transitions more heavily during sampling, ensuring the agent revisits transitions where it made the most incorrect predictions. Combined with D-DQN, this brought the validation Hit-Rate from 0.298 (D-DQN with MSE) to 0.614 (D-DQN + SmoothL1 + prioritized replay + soft updates) — the best the modified point localizer achieved before switching to the full ResNet-50 extractor.

### Replacing the inbuilt feature extractor with ResNet-50

The reference implementation trains a 5-layer CNN feature extractor alongside the DQL task, forcing the network to learn image representations and localization decisions simultaneously. This extends training time significantly and was identified as a likely factor in slow convergence.

Using a pretrained ResNet-50 (frozen) as the feature extractor separates these concerns: the encoder provides general visual features; only the FC decision layers are trained on the DQL task. This reduced trainable parameters by approximately 28% and produced the best accuracy across all configurations (0.892 vs 0.805 for the custom extractor).

### Bézier curve selection: 3 control points, not 4

The initial curve design used 4-control-point Bézier curves. The action space for a 4-point curve — moving start, end, and two curvature points in each direction, then terminating — results in 27 actions per control-point adjustment cycle. For a DQL agent operating on discrete actions, this is large. Reducing to 3 control points (start and end fixed; only the middle curvature point adjusted) collapsed the action space to 9 moves while retaining the curve's ability to represent non-straight lanes.

## Algorithm & Training Design

### Point localization game

The agent controls one landmark point at a time, iterating through all detected points on a lane sequentially. For each point, the state is the concatenation of the ResNet-50 feature vector from the lane crop and the current pixel position of the point. The agent selects from 4 actions: move left by 5 pixels, move right by 5 pixels, delete (move to x₀−20), or terminate.

Rewards are distance-based: a regular move earns +1 if it reduced pixel distance to ground truth and −1 otherwise. Terminal actions earn +3 if the final distance is within 5 pixels of ground truth, and −3 otherwise. Invalid actions earn −5. The negative terminal reward for landing outside the goal threshold is the key learning signal — it teaches the agent to localize accurately rather than simply terminate early.

### Curve localization game

The `CurveAgent` replaces individual points with a 3-control-point Bézier curve. Lane points are converted to Bézier control points before the game; after each move, the control points are rendered back to lane points for distance computation against the ground truth (which remains in point format). The start and end control points are fixed; only the curvature control point is moved.

The 9-action space covers 2D directional moves (diagonal and cardinal directions at 20-pixel steps) plus terminate. The goal distance threshold is relaxed to 25 pixels to account for the coarser representation.

### Training configuration

All experiments used Adam (lr=1e-4, weight_decay=5e-4), discount factor γ=0.9, replay buffer capacity 104,200, batch size 32, 1,000 warm-up steps, and a maximum of 100,000 training steps with validation every 10,000 steps. Epsilon decayed exponentially from 1.0 to 0.05.

## Results

<Callout title="Core finding: DQL localization did not improve on the CV detector">
  Across all configurations — custom extractor, ResNet-50 extractor, Bézier curves — the DQL localizer produced accuracy at or below the initial CV detection baseline. The hypothesis that iterative DQL refinement would improve precision was not supported on TuSimple.
</Callout>

| Configuration | Hit-Rate | acc | FPR | FNR |
|---|---|---|---|---|
| CV Detector (baseline) | 0.784 | 0.900 | 0.233 | 0.220 |
| Point Localizer (custom extractor) | 0.610 | 0.805 | 0.449 | 0.422 |
| Point Localizer (ResNet-50, no history) | 0.626 | **0.892** | 0.239 | 0.226 |
| Point Localizer (ResNet-50, 8-frame history) | 0.612 | 0.888 | 0.245 | 0.233 |
| Bézier Curve Localizer | 0.590 | 0.783 | 0.452 | 0.446 |

The best configuration — ResNet-50 extractor without history — reached acc 0.892. The CV detector baseline is 0.900. The localization step degraded accuracy by approximately 1pp on this metric, while FPR and FNR remained similar to the baseline. Every other configuration degraded accuracy more severely.

### Why the point localizer underperformed

The detect Hit-Rate (fraction of points already within the 5-pixel goal threshold before any agent action) was 0.731 in early experiments. The replicated base DQL reached a peak Hit-Rate of only 0.65 at 100,000 steps — worse than leaving the initial detection untouched. After all fixes, the best Hit-Rate reached 0.626. The localization game is not producing net improvement: some points are moved closer to ground truth, but others are moved further away, and the net effect is degradation.

One structural reason: the DQL agent operates on one point at a time, without a global view of the lane. A point that is already within the goal threshold may still be moved by the agent before it terminates. With up to 50 points per lane, the agent must make 50 sequential decisions correctly — each wrong terminal action or unnecessary move degrades the reconstructed lane.

### Why the curve localizer underperformed further

The Bézier curve localizer stagnated in training — the validation Hit-Rate showed no improvement after the initial steps. The most likely cause: lanes in TuSimple are predominantly straight, so the initialization curve (a straight line connecting the detected endpoint positions) is already a close approximation of the ground truth. The agent learned to terminate quickly without adjusting curvature. In effect, the curve representation provided no advantage on a dataset where straight-line initialization is already near-optimal, while the expanded 9-action space made exploration harder.

Visualizations confirmed this: predicted curves appeared as straight lines across nearly all test cases, regardless of ground truth curvature.

## Constraints & Limitations

<Callout title="Time and compute constrained the experiment scope">
  100,000 training steps on a single RTX 4000 with image processing at 224×224 limited the number of configurations tested. Extensive hyperparameter tuning was not feasible. The results establish whether the approach works under standard settings, not whether it could work with significant tuning.
</Callout>

**TuSimple's lane distribution limits curve evaluation.** TuSimple is a highway dataset with predominantly straight lanes. The curve localizer was designed to handle curved lanes; testing it primarily on straight lanes means the hypothesis (curves help with curvature) was not fully tested. CULane, which includes urban and curved scenarios, would be a more appropriate evaluation dataset.

**The action granularity was not tuned per dataset.** Both the 5-pixel point step and the 20-pixel curve step were adapted from the reference paper or set by heuristic. The optimal step size depends on image resolution and lane geometry — neither was systematically investigated.

**History frames did not help; normalization was not applied.** The standard practice in DQL implementations is to normalize Q-values and action values before feeding them into the history vector. This was not done here. The marginally worse performance of the 8-frame history configuration (acc 0.888 vs 0.892) likely reflects training instability from unnormalized inputs, not a fundamental problem with temporal history.

**DQL may not suit dense sequential localization.** The core structural challenge is that DQL was designed for sparse decision problems (one action per global state) and is being applied here to dense sequential prediction (one action per lane point, repeated 50 times per lane). This mismatch may be more fundamental than any implementation detail.

## Next Steps

The results suggest the framing matters as much as the algorithm. Several concrete directions:

- **Evaluate on CULane or custom curved-lane datasets.** TuSimple's straight-lane dominance makes curve localization hard to evaluate fairly. A dataset with more curved lanes would test whether the Bézier hypothesis holds where it was intended to.
- **Global lane action space.** Rather than one decision per point, train an agent to output adjustments for all lane points in a single step. This removes the compounding error from sequential per-point decisions and gives the agent a holistic view of the lane.
- **Reduce the point-per-lane count before training.** Thinning from 50 points to 6–8 representative positions would reduce sequential decisions per lane, allowing the agent to process more lanes within the same step budget.
- **Normalize action history inputs.** Standard DQL practice; likely to stabilize the history-enabled configuration and allow a fairer test of temporal context.
- **Explore policy gradient methods.** Algorithms like PPO handle continuous action spaces and can optimize over a sequence of moves jointly, which may better match the lane localization problem structure than per-point DQL.

The fundamental question — whether a learned post-processing agent can improve on one-shot lane detection without retraining the detector — remains open. This study establishes what doesn't work with point-by-point DQL on TuSimple; it doesn't rule out the broader approach.

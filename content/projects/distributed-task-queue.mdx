## Detailed Problem

The client processed incoming data events from a SaaS platform in batches via a cron job. As data volume grew, the cron approach became fragile: jobs would pile up, latency spiked unpredictably, and a single failing job could block the entire queue with no visibility into why.

The constraints were strict: no changes to the upstream data source, a two-week delivery window, and a requirement to run on existing Kubernetes infrastructure.

## Architecture

<Callout type="tradeoff" title="Redis vs. purpose-built queue">
Using Redis as a queue backend keeps the infrastructure footprint small — no Kafka or RabbitMQ cluster to operate. The tradeoff is that durability depends on Redis persistence configuration. For this client's SLA (best-effort, no strict durability guarantee), it was the right call.
</Callout>

The system has three components:

- **FastAPI control plane** — job submission, priority assignment, status polling
- **Redis queue** — ZSET-backed priority queue per job type
- **Worker pool** — horizontally scalable Kubernetes Deployment consuming jobs

## Results

At steady state under synthetic load matching 3× the client's peak traffic, P99 scheduling latency was 87ms. Zero jobs were lost during a 30-day production run, including one Kubernetes node failure that self-healed within the retry window.

## Next Steps

The priority system currently supports three fixed lanes. Dynamic lane weights — adjusted based on queue depth and SLA targets — would give the client more control over throughput shaping.

## Detailed Problem

A résumé compresses years of engineering into a one-page document built for a non-technical first reader, and a LinkedIn profile does the same. Both flatten the dimension that actually separates engineers — how they reason through a hard problem — into keyword lists and job titles. Anyone judging real technical depth has to go digging for it — through the reports, the repositories, the commit history.

A portfolio site recovers some of that lost depth: it is long-form space to show the work directly. The agent building it here is Claude Code, a coding agent driven by a large language model (LLM) that reads and edits files, runs commands, and operates a repository under instruction.

The goal was a portfolio that is credible and deep, and built the way the rest of the work is built — by an agent, under explicit engineering discipline, leaving an inspectable trail rather than a black box. A second goal made it harder: the machinery had to be reusable, so the same system could seed later projects instead of being discarded after one.

The difficulty is not getting an agent to produce code. It is keeping its output consistent and trustworthy across hundreds of edits. Agents drift from intent, over-produce, and report success without evidence. A system that ships a public artifact has to constrain those failure modes — hold a quality bar, keep every change reviewable, and make every claim verifiable — without a human re-reading every line.

## Approach

The organizing idea is that building software with an agent is an orchestration problem, and the specification is the contract that governs it. The agent is capable but unanchored; the specification is what ties it to intent.

That contract has three layers. **Documentation** defines what to build — product intent, the design system, the content schema, the build plan. **Rules** constrain how the agent works — engineering standards, git discipline, accessibility and design-system constraints, and the verification required before work is called done. **Skills** encode repeatable capabilities — a procedure the agent runs the same way every time, such as turning a project into a portfolio page.

Two simpler approaches don't clear the bar. Ad-hoc prompting produces a different result every run and leaves no reusable discipline behind. A template or generic site generator produces a site but no engineering narrative, and nothing that transfers to the next project. Treating the specification as a contract trades up-front specification effort for consistency and reuse — the property the other two approaches lack.

## Architecture

The contract is realized as a layered set of files the agent reads before acting. Each layer answers a different question, which keeps them independently editable and lets their constraints compose.

- **Documentation** — the specification: product, design-system, content-schema, and implementation-plan documents. A precedence order resolves conflicts so the agent never guesses which instruction wins — the design document is authoritative on visual questions, the schema document on content fields.
- **Rules** — tiered constraints: stack-agnostic engineering and git discipline at the base, then web-frontend, framework, and project-specific tiers above it. The agent loads only the tier relevant to the file it is touching.
- **Skills** — self-contained capability procedures, each with its own references and quality bar: content extraction, asset generation, cover generation, and review.
- **Pipelines** — skills composed into a fixed sequence for a unit of work.
- **Gates** — the checkpoints a change must clear before it advances: plan approval, build verification, automated hooks, and human sign-off.

The separation is the point. What to build, how to build it, what the agent is capable of, in what order, and what is allowed to pass are five different concerns, each owned by one layer. A change to the design system touches one document; a new capability is one skill; a stricter quality bar is one gate — none of them disturb the others.

## Workflow

Architecture is the static structure; the workflow is how it runs. Three loops operate together.

**The build pipeline.** A project page is produced by four skills in a fixed order — content extraction, asset generation, cover generation, and review. Each stage hands off only after the human approves its output, so a mistake is caught at the stage that produced it instead of at the end.

**The plan-and-gate loop.** Any non-trivial task starts in plan mode: the agent proposes a plan and waits. No file is written until the plan is approved, and the repository index is never staged without an explicit go-ahead. A task is complete only when verification has run and the human has signed off — a passing build is necessary but never sufficient. A status question halts the agent rather than starting work.

**The self-improvement loop.** When the same correction recurs, it is recorded in a lessons file as a rule the agent consults on later tasks. The loop is deliberately manual: a written correction, not an automated reward signal.

## Engineering Decisions

### Specification precedence

Conflicting instructions are resolved by a defined order rather than left to the agent's judgment. The design document wins on visual questions, the schema on content fields, and project rules over general ones. Precedence removes a whole class of silent, inconsistent decisions — when two sources disagree, the outcome is determined, not guessed.

### Cold two-pass review

The review stage runs two independent reads — one from a recruiter's perspective, one from a technical hiring manager's — as separate subagents with no shared context and no sight of each other's notes. The point is the cold read: an author cannot reliably catch what they are blind to, and a reviewer who has seen the author's reasoning inherits the same blind spots. Isolating the passes is what makes the review adversarial rather than confirmatory.

### Deterministic, reproducible release

The published site is a separate branch with an orphan history that shares no commits with the development branch, snapshot-replaced from an allowlist on every release and never merged from it. Identical inputs produce an identical published tree, and nothing outside the allowlist — development history, planning documents, the system files themselves — can leak into what ships. The live site is a build artifact, not a branch someone edits by hand.

### The hidden-system trade-off

The system files live in the repository's working tree but are deliberately untracked. A commit guard blocks committing them, and the reusable core is meant to live in its own home rather than be coupled to one project. The alternatives were both worse: tracking the files fights the guard and binds the system to this repository, while deleting them from the working tree discards the operating context the agent needs. The accepted cost is that the files steer every build but are absent from the public history — so the work is verifiable through the commit trail of what they produced, not by reading the files directly.

## Outcomes

The system delivered a live, statically generated portfolio. Every route is prerendered to static HTML at build time; there is no server-rendered or client-fetched content path to slow the first paint or fail without a running server.

The verification is built into the constraints rather than bolted on afterward, and each property traces to a mechanism rather than a final cleanup pass:

- **Performance** — prerendering every route to static HTML at build time removes the server round-trips and client fetches that usually cost the first paint; that is what the desktop Lighthouse score reflects.
- **Accessibility** — contrast is enforced at the design-token level, so a palette change is checked once against WCAG AA across both themes instead of being re-audited component by component.
- **Security** — a Content Security Policy (CSP) defined in one place blocks framing, foreign objects, and cross-origin script beyond the framework's unavoidable inline bootstrap, and the hardening headers ship with it from the same configuration.
- **Machine readability** — the sitemap, robots policy, JSON-LD structured data, and llms.txt summary are generated from the same content source as the pages, so they stay in sync with what actually ships.

The reusable core behind the site was distilled across three projects — this portfolio and two machine-learning projects — and is being consolidated into a standalone, installable agent-orchestration system, Claude OS. Only the portfolio needed its own bucket of project-specific rules and skills; the shared engineering discipline seeded the rest.

This case study is itself an output of the system it describes. The page was produced by the content pipeline, held to the same rules, and stopped mid-write by the same automated checks that guard every other change — its construction is legible in the repository's commit history.

## Limitations

The discipline has a cost. The gates and plan-mode add latency: a change a developer would make in minutes can take several approval cycles. That trade is worth it for a public artifact and pure overhead for throwaway work — the system does not try to be the right tool for both.

It is also not autonomous. Every stage is gated on human direction and approval; the agent executes within the lines but does not draw them, and output quality tracks the quality of the steering it is given.

The scope was kept deliberately narrow. Web-development and visual-design depth were routed to a planned blog instead of this page, so this is an account of the build system, not of the front-end engineering underneath it — a scope choice, not an omission.

Finally, the self-improvement loop only ever ran by hand. The half meant to score and tune the system's own skills automatically did not run in this toolchain, so improvement stayed a manual practice: corrections recorded deliberately, not learned from measured outcomes.
